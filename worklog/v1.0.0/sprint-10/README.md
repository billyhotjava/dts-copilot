# Sprint-10: 园林业务语义层与 NL2SQL 落地 (BG)

**前缀**: BG (Business Grounding)
**状态**: READY
**目标**: 为 `dts-copilot` 建立面向 `adminapi/adminweb` 的业务语义层，以"业务视图层 + 预制查询模板 + 规则路由"为核心架构，优先打通项目履约与现场业务两个主题域的 NL2SQL 能力。

## 背景

代码审查已经确认：

- `adminapi/adminweb` 的主业务轴集中在 `project / flowerbiz / tasknew / pendulum`
- `dts-copilot` 已具备外部业务库接入、表字段元数据、同义词、指标对象、NL2SQL eval case 等底座
- 当前问题不在"能不能查库"，而在"能不能稳定理解业务、走对口径、受权限约束地执行"

## 方案核心变更（相比初版）

基于对 `adminapi/adminweb/app/doc` 四个目录源码的深度审查，识别出五个 NL2SQL 核心难点：

1. **状态码地狱** — 同名字段 `status` 在不同表含义完全不同（`p_project`: 1=正常; `t_flower_biz_info`: 1=审核中）
2. **业务类型编码** — `biz_type` 10+ 种含义，"报花"是统称不是某一种类型，"减花"≠"剪花"
3. **时间字段歧义** — 单表 6+ 时间字段（`apply_time`/`create_time`/`finish_time`/`start_lease_time`），用户说"上个月的加花"指哪个？
4. **结算逻辑不可 SQL 化** — 两种计费模式 + 折扣率 + 结算周期 + 跨月分摊，写在 Java Service 层
5. **多表 join 组合陷阱** — 同一主语不同问题走完全不同的 join 路径（客户绿植 vs 客户账单 = 完全不同的 4 表 join）

因此本 Sprint 不再以"语义包 + join contract + ELT 主题层"为核心架构，改为：

- **业务视图层**：预 join + 状态码翻译，模型只查视图不查原始表
- **预制查询模板**：高频 TOP 20 问句直接走参数化 SQL，准确率 100%
- **规则路由**：关键词 + 分类器判定业务域，不消耗 LLM 调用
- **枚举词典**：所有状态码、业务类型外置为可查询的词典表

## 交付范围

### 主题域

- 项目履约：项目、客户、合同、点位、项目绿植、履约任务
- 现场业务：加花、减花、换花、转移、回收、初摆、执行任务

### 运行时架构

```
用户打开 Copilot 面板（BG-11 欢迎卡片 + 推荐问句）
  │
  ├─ 用户输入问句（或点击推荐问句）
  │
  ├─ 预制模板匹配（BG-04）
  │    ├─ 命中 → 参数提取 → 参数化 SQL → analytics 执行 → 返回
  │    └─ 未命中 ↓
  │
  ├─ 意图路由（BG-07，规则引擎）
  │    ├─ 识别业务域 → 选择目标视图
  │    ├─ 低置信度 → 澄清
  │    └─ 高置信度 ↓
  │
  └─ NL2SQL 生成（现有 Nl2SqlService）
       ├─ 上下文 = 目标视图 DDL + 枚举词典 + 同义词 + few-shot
       ├─ 只允许查视图，不允许查原始表
       └─ 权限桥接（BG-09）→ analytics 执行 → 返回
                                                    │
  ┌─────────────────────────────────────────────────┘
  │
  └─ Chat 内 SQL 预览 + 内联执行（BG-11）
       ├─ SQL 预览卡片（可编辑）→ 内联执行 → 结果表格
       ├─ "创建可视化" → 跳转 CardEditor
       └─ 反馈 👍👎 → ai_chat_feedback → 数据飞轮
```

### 关键设计原则

- **模型只查视图**：视图中预 join + 状态码翻译中文，消除 join 陷阱和编码歧义
- **预制模板优先**：高频问句 100% 准确率兜底
- **规则路由优先**：关键词匹配判定业务域，不消耗 LLM
- **结算直查结果**：结算类问题查 `a_month_accounting` 已计算结果，不让模型重算
- **交互闭环**：推荐问句降低使用门槛 → 内联执行消除跳转断裂 → 反馈回收驱动迭代
