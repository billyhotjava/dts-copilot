# T02: 报花全生命周期状态机文档化

**优先级**: P0
**状态**: READY
**依赖**: 无

## 目标

把 7 状态 + 13 bizType 的完整业务语义文档化（不仅是字段值，而是"每个状态意味着什么、谁触发、停留太久意味什么、到下一状态的条件"）。F1 字典 / F4 dim 表 都基于此 task 输出物。

## 技术设计

### 1. 7 状态完整状态机

为每个状态回答这 6 个问题：

| 问题 | 示例（status=1 审核中）|
|---|---|
| 业务含义 | 申请人提交后，等待审批人审核 |
| 谁触发进入 | 申请人（养护人 / 项目经理 / 业务经理） |
| 谁触发离开 | 审批人 |
| 离开的可能去向 | → 2 备货中（通过）/ → 21 驳回（退回）/ → -1 作废（取消） |
| 停留过久意味什么 | 审批积压；> 3 天 = 异常；> 7 天 = 严重 |
| 关联表写动作 | `t_flower_biz_info.status` UPDATE + `t_flower_biz_info.examine_user_id` SET |

完整覆盖 7 状态：-1 / 1 / 2 / 3 / 4 / 5 / 20 / 21（8 个，含变体）。

### 2. 13 bizType 完整业务语义

为每个 bizType 回答这 8 个问题：

| 问题 | 示例（biz_type=2 加花）|
|---|---|
| 业务术语（口语） | 加摆 / 加植 |
| 业务术语（正式） | 加花 |
| 触发场景 | 客户增订单、新摆位添置、季节更换 |
| 触发角色 | 养护人 / 项目经理 |
| 金额方向 | +正（rent 增加） |
| 关联模块联动 | 触发 PlanPurchaseItem → DeliveryInfo → WarehousingInfo |
| adminweb 入口 | `flower/flowerbiz/add/list-add-flower.vue` |
| 完成态行为 | p_project_green 加新行；客户结算应收增加 |

完整覆盖 13 bizType：1/2/3/4/6/7/8/10/11/12（10 个有效 +3 待识别）。
