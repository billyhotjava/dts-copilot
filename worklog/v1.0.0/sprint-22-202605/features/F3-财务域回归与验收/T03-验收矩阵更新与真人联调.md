### 2. 真人联调清单

#### A. 固定报表验收（8 张）

```
1. 打开 https://ai.yuzhicloud.com/reports/fixed
2. 依次执行：
   - FIN-AR-OVERVIEW（应收概览）
   - FIN-CUSTOMER-AR-RANK（客户欠款排行）
   - FIN-PROJECT-COLLECTION-PROGRESS（项目回款进度）
   - FIN-PENDING-RECEIPTS-DETAIL（待收款明细）
   - FIN-PENDING-PAYMENT-APPROVAL（待付款审批）
   - FIN-ADVANCE-REQUEST-STATUS（预支申请状态）
   - FIN-REIMBURSEMENT-STATUS（报销状态）
   - FIN-INVOICE-RECONCILIATION（发票对账）
3. 每张报表确认：
   - 200 响应
   - 字段中文 label
   - 状态字段显示中文（如"已结算"而非数字 2）
   - 金额格式正确（保留 2 位小数）
   - 与旧系统对应页面口径一致（数值不偏差 > 1%）
```

#### B. NL2SQL 问句验收（12 条）

```
打开 ChatPanel，依次输入：

1. 客户欠款前 10 是谁？
2. 本月各项目应收多少？
3. 2025 年 3 月哪些项目还没收款完？
4. 万象城那个项目 2025 年回款进度
5. 待付款审批里有没有超过 30 天的？
6. 上月预支单还有多少没核销？
7. 2025 年 Q1 总开票金额
8. 张三的报销单到哪一步了？
9. 客户 X 还欠多少？（X 用真实存在的客户名）
10. 应收账款情况
11. 项目租金（边界：应路由到 project，不到 finance）
12. 采购金额前 10（边界：应路由到 procurement，不到 finance）

每条问句记录：
- 是否命中 TEMPLATE_FAST_PATH（< 1s）
- 路由日志中显示的 domain / template_code
- 生成 SQL 中有无错链
- 结果与旧系统对照
```

#### C. 字段与字典验收

```
C.1 字段中文 label
- 打开 /questions 创建新查询
- 数据源选 rs_cloud_flower
- 表选 authority.finance.settlement_summary
- 字段选择器中确认显示：账期 / 应收金额 / 已收金额 / 待收金额 / 项目点 / 客户 等中文 label

C.2 字典翻译
- ChatPanel 输入：哪些结算单是已结算状态？
- 确认生成的 SQL 中状态条件可读（中文标签或注释）
- 结果中状态列显示中文标签
```

#### D. 边界与 Guardrails

```
D.1 跨域不冲突
- ChatPanel 依次输入：
  - 项目租金 → 期望：project 域
  - 采购金额前 10 → 期望：procurement 域
  - 客户欠款前 10 → 期望：finance 域
  - 项目应收 → 期望：finance 域
- 三个域规则不互抢

D.2 Guardrail case 不复现
- 对照 finance-guardrail-cases.md，逐条问 8 个失败 case
- 确认 LLM 生成的 SQL 不再走错链
```

#### E. 双轨衔接（F4 完成后执行）

```
E.1 datasource 注册可见
- 打开 /admin/datasources
- 确认存在两条财务相关 datasource：
  - 旧：sprint-21 PG 视图（authority.finance.*）
  - 新：花卉财务 Mart (dbt)，指向 dts-stack dbt 产出库

E.2 ~ E.5 同问题切 datasource 结果一致
- 在 ChatPanel 选 datasource = 旧，问 "客户欠款前 10"，记录前 5 客户名 + 金额
- 切到 datasource = 新，问同样问题
- 期望：前 5 客户名完全一致，金额差 < 0.01 元
- 重复 5 条代表性问句

E.6 sprint-23 占位（本 sprint 不验）
- 待 dts-copilot 接入 OpenMetadata Glossary 后再验收
- 验证：finance 域同义词通过 OM API 派生，与 finance.json 一致
```
