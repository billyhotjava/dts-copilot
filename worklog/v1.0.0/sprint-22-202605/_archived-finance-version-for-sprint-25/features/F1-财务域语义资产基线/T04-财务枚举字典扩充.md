# T04: 财务枚举字典扩充（短期方案）

**优先级**: P0
**状态**: READY
**依赖**: T01

## 双轨说明

本 task 是**短期方案**：直接 INSERT `BizEnumDictionary`，让 finance 域问句立即可用枚举翻译。

**长期由 F4-T04 替代**：F4 完成后，状态枚举沉淀到 OpenMetadata Glossary，dts-copilot 启动时通过 OpenMetadata API 拉取最新词汇；本 task 写入的字典数据成为 fallback 兜底。

特别地，dbt 模型层（dwd）会用 `dim_*_alias` 维度表对状态字段做翻译（参考 dts-stack 已有的 `dim_quality_status_alias` / `dim_risk_level_alias` 等模式），即"在数据层就把数字 → 中文"。届时：

- 视图层字段 `*_status_name` 已是中文 → 不再依赖字典翻译
- 字典仅用于"问句中提到中文标签 → 反向解析回字段值"以及"LLM 走探索路径需查物理表"的场景

## 目标

把 adminapi `rs-flowers-base/src/main/java/com/rs/flowers/base/finace/` 与 `expense/` 的状态枚举系统化沉淀进 `BizEnumDictionary`，让 LLM 在写 WHERE / 输出结果时使用中文标签而不是数字编码。

### 1. 字典扫描清单

### 3. 实际值与文档值的差异

> ⚠️ 上表是**草稿**。T04 实际执行时必须先用以下命令校对真实数据：
> 
> ```sql
> SELECT DISTINCT status FROM f_settlement_info;
> SELECT DISTINCT status, pay_type FROM f_pay_record;
> SELECT DISTINCT status FROM t_advance_info;
> SELECT DISTINCT status, category FROM a_expense_account_info;
> SELECT DISTINCT status FROM a_invoice_info;
> ```
> 
> 查到的实际值**优先于本文档**。文档仅作为初始采集模板。

每条记录形如：

### 5. 用法对接

- `Nl2SqlService` 在 prompt 注入时，把字典翻译成 `WHERE settlement.status = 2  -- 已结算` 注释，让 LLM 看得懂
- `BizEnumService` 提供 `translate(enumType, value) -> label` 与 `reverseTranslate(enumType, label) -> value`，结果 SQL 输出时把数字翻译成中文
- 视图层已经做了翻译的字段（如 `settlement_status_name`），LLM **不**再走字典 —— 本字典仅服务"问句中提到中文标签 → 转回字段值"以及"问句指向底层物理表（探索模式）时翻译"
