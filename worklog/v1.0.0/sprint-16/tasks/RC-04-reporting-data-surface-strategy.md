# RC-04 受控取数面策略（L0 / L1）

**状态**: DONE
**目标**: 为固定报表快路径设计可落地的取数策略，避免已知报表继续走“全库自由 NL2SQL”。

## 分层

### L0 业务权威直读

适用：

- 状态台账
- 明细台账
- 待办
- 实时状态查询

示例：

- 财务结算列表
- 开票管理
- 日常报销
- 预支申请
- 采购明细
- 入库/出库台账

### L1 准实时主题层

适用：

- 趋势图
- 排行
- 跨实体汇总
- 时间序列分析

示例：

- 财务结算汇总
- 月度财务报表
- 采购汇总
- 库存趋势
- 项目点汇总

## 当前约束

- `rs_cloud_flower` 无视图层
- `dts-copilot` 当前 analytics 只有轻量 `mart/fact`
- `0036_business_views_metadata.xml` 登记的 `7` 个 `v_*` 业务视图目前只存在 SQL 资产，不在测试库中
- `0040` 固定报表种子中的 `authority.*` / `mart.finance.*` / `fact.procurement.*` 目标大多还没有真实 backing

## 建议

第一阶段不强推完整 `DWD/DWS/ADS`，而是：

- 在 analytics 内补最小可用的财务/采购/仓库主题资产
- 同时保留 L0 业务权威直读
- 用模板决定一张固定报表走 L0 还是 L1

## V1 具体策略

### L0 业务权威直读

第一阶段应优先覆盖这些现网页面：

- 财务结算列表
- 开票管理
- 日常报销
- 预支申请
- 采购明细
- 采购计划明细
- 采购驳回
- 配送记录
- 库存现量
- 出入库记录
- 入库管理
- 出库管理
- 报损管理
- 调拨管理
- 调拨明细
- 退货管理

这类页面的共同点是：

- 粒度清晰
- 现网页面已经稳定
- 适合直接复用 `adminapi` 现成控制器/导出能力
- 不应继续等 MySQL 视图层落地

### L1 准实时主题层

第一阶段应优先新增或补齐以下主题资产：

- `mart_finance_settlement_daily`
  - 对应：财务结算汇总、财务报表（月度）
- `mart_procurement_summary_daily`
  - 对应：采购汇总、采购计划汇总、供应商采购排行
- `fact_inventory_movement_event`
  - 对应：出入库记录、调拨、报损、退货
- `mart_stock_snapshot_daily`
  - 对应：库存趋势、低库存预警、仓库概览
- `mart_project_summary_daily`
  - 对应：项目点汇总、实摆物品统计、实摆租金统计

## 不再继续使用的错误假设

1. 不能假设 `rs_cloud_flower` 已有可直接使用的 MySQL 视图
2. 不能假设 `authority.finance.*` 就等价于真实权威取数面
3. 不能让固定报表目录继续种入与现网页面无直接映射关系的抽象模板
