# RC-01 现网页面盘点与报表型页面识别

**状态**: DONE
**目标**: 基于真实登录态和 `adminweb/app` 源码识别现网系统里真正的“报表型页面”和移动端实际覆盖，避免只看一级菜单得出错误结论。

## 结果

- 当前账号可见路由：`67`
- 其中报表型页面：至少 `60`

## 关键发现

### adminweb

PC 端高密度业务目录如下：

- `flower/purchase` `13` 个页面
- `flower/store` `26` 个页面
- `flower/finance` `22` 个页面
- `flower/operate` `20` 个页面
- `flower/project` `46` 个页面
- `flower/task` `51` 个页面
- `plant` `23` 个页面

### app

移动端高密度目录如下：

- `pages/task` `100` 个页面
- `pages/flowers` `88` 个页面
- `pages/purchase` `20` 个页面
- `pages/warehousing` `13` 个页面
- `pages/pendulum` `17` 个页面
- `pages/project` `8` 个页面

结论：

- PC 端是固定报表主载体
- 移动端更适合作为任务、报花、采购、仓库的查询和操作补充面

## 结论

现网系统不是“缺少报表能力”，而是“报表能力未被统一目录化和模板化”。后续应优先将这些现有业务页沉淀成固定报表模板，并把移动端限制在适合的轻量查询和执行场景，而不是重新从零生成。
