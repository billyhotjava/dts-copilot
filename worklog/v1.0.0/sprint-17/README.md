# Sprint-17: 主数据优先治理与业务锚点收口 (MD)

**前缀**: MD (Master Data)
**状态**: DONE
**目标**: 基于 `adminapi / adminweb / app` 三端源码和 `rs_cloud_flower` 业务库，先建立“项目轴 + 物品轴”优先的主数据治理框架，明确主数据、参考数据、配置型事实和交易事实的边界，并为固定报表与 Copilot 的后续受控消费提供稳定骨架。

## 背景

对现网系统和三端源码的复核表明：

- 业务主线不是标准 ERP，而是“项目点履约 + 报花业务 + 采购仓配 + 业务财务”
- 后端虽然分了 `project / flowerbiz / tasknew / purchase / storehouse / finace / operate` 等目录，但这些更像业务文件夹，而不是清晰的领域边界
- 前端大量页面本质上是报表型页面，但散落在业务菜单里
- 业务库 `rs_cloud_flower` 有 `163` 张物理表、`0` 张视图，说明系统更依赖业务表直出，而不是稳定的数据消费面

因此，如果直接从财务报表或 Copilot 下手，会持续受到上游对象定义不一致的影响。  
更合理的切入点，是先把主数据锚点收住。

## 核心判断

### 推荐路径

不是“先做财务主数据”，而是先做：

1. 项目轴主数据
2. 物品轴主数据
3. 共享参考主数据
4. 财务参考主数据

### 为什么不是财务先行

当前系统里的财务更多是业务财务，不是独立财务中台：

- 账单和对账偏 `operate`
- 凭证和流水偏 `finace`
- 预支和冲抵穿插在 `advance`、流程和付款链

这意味着如果不先统一项目、物品、采购、库存等上游锚点，财务口径仍然会漂。

## 主数据优先级

### P0：业务锚点主数据

- 客户
- 合同
- 项目点
- 楼层 / 楼号 / 摆位
- 物品
- 分类
- 属性
- 规格
- 价格
- 别名

### P1：共享参考主数据

- 库房
- 供应商
- 组织 / 员工 / 角色
- 状态字典
- 来源码 / 结算方式 / 付款方式

### P2：财务参考主数据

- 科目
- 账期
- 发票类型
- 冲抵类型
- 付款类型

## 当前已识别的典型问题

1. 页面、控制器和数据库归属不一致  
2. 状态码、来源码、字典值分散在前端页面和服务实现里  
3. 供应商不是清晰的一等主数据  
4. 项目绿植、合同租金规则等“配置型事实”容易被误当主数据  
5. 库存余额、采购汇总、账单、凭证等“交易/派生事实”反向影响主数据口径

## 预期产出

1. 一套明确的主数据对象树
2. 一套主数据 vs 配置事实 vs 交易事实的边界表
3. 一套主数据消费规则，约束固定报表和 Copilot
4. 一份后续逐步实现的执行基线，而不是大爆炸重构

## 验收标准

- 每类主数据都有唯一的业务归属
- 项目轴和物品轴具备稳定业务键与展示名
- 供应商、库房、组织、字典的治理优先级明确
- 固定报表和 Copilot 的消费面能明确依赖哪些主数据
- 后续实现可以按任务逐步推进，不需要推翻现有业务系统
