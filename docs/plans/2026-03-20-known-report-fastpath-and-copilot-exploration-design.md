# 已知报表快路径与 Copilot 探索兜底设计

## 背景

在客户演示中，当前 `dts-copilot` 的主要问题不是“不能查”，而是：

- 已知高频报表没有固定入口，仍然大量依赖大模型现场找库、找表、拼 SQL
- 财务、采购、仓库等高敏感场景对口径稳定性和响应速度要求更高
- 现网系统本身是综合业务平台，重心不在 BI 平台，而在：
  - 报花管理
  - 任务中心
  - 项目点管理
  - 运营管理
  - 采购管理
  - 仓库管理
- 其中客户优先级进一步明确为：
  - 财务相关第一位
  - 采购相关第二位
  - 其他业务域随后推进

当前 `dts-copilot` 已具备以下底座：

- 外部业务库注册与元数据同步
- 业务语义视图层 `v_*`
- 轻量主题层 `mart_project_fulfillment_daily` / `fact_field_operation_event`
- dashboard / screen / report factory / copilot chat
- semantic pack / templateCode / eval / feedback

但当前产品主路径仍偏“探索型 Copilot”，不适合作为已知业务报表的主入口。

## 设计目标

1. 让已知高频报表不再依赖 LLM 现场生成 SQL。
2. 让财务、采购相关报表优先具备产品级响应速度与口径稳定性。
3. 保留未知问题的探索能力，但把它限制在受控语义面内。
4. 将未知高频问题持续沉淀成新的固定报表模板。
5. 统一 dashboard / screen / report factory / copilot 的报表资产，不再各自维护一套。

## 非目标

- 本阶段不建设完整 `ODS/DWD/DWS/ADS` 数仓体系
- 本阶段不让所有报表都承诺绝对实时
- 本阶段不让 Copilot 直接替代固定报表目录
- 本阶段不把财务模块重构成总账型 BI 产品

## 核心判断

### 1. 已知报表与未知报表必须分流

如果已知报表仍然走 `NL2SQL -> 实时查库 -> 动态渲染`，则：

- 响应不稳定
- 口径难以固化
- 演示效果差
- 高敏感财务/采购问题风险高

因此必须拆成：

- 已知报表快路径
- 未知报表探索路径

### 2. 财务与采购必须优先走固定模板

财务和采购类问题往往涉及：

- 应收 / 已收 / 未收
- 待付款 / 付款状态
- 采购在途 / 待入库 / 库存可用
- 供应商金额排行 / 采购时效 / 库存预警

这些问题要优先保证：

- 口径固定
- 权限清晰
- 响应稳定

因此不能以开放式 Copilot 作为主路径。

### 3. 实时性要分层，不应一刀切

“最好实时”在业务上成立，但技术上不应等同于“所有问题都查实时事实表”。

建议拆成三档：

- `T0 实时`
  - 状态类、明细类、待办类
  - 直接走权威视图或权威查询
- `T1 准实时`
  - 趋势、排行、跨月统计
  - 1 到 5 分钟刷新主题层或缓存
- `T2 离线/固化`
  - 月结、正式经营分析、关账后口径
  - 按结算周期生成

## 总体架构

```text
用户问题 / 报表入口
  -> 模板命中层
      -> 命中固定模板 -> 固定报表快路径
      -> 未命中 -> Copilot 探索路径

固定报表快路径
  -> ReportTemplate
  -> 权威视图 / 主题表 / 缓存
  -> Dashboard / Screen / Report Factory / Catalog

Copilot 探索路径
  -> 业务域识别
  -> 受控视图/主题层选择
  -> NL2SQL / 查询计划
  -> 结果分级返回
  -> 候选模板池
```

## 报表产品分层

### L0 权威明细层

用于实时明细和状态类报表：

- 财务待收/待付明细
- 预支/报销/付款单状态
- 采购申请/采购单/配送/入库状态
- 库存现量/低库存

特征：

- 直接读取业务权威视图或受控 SQL
- 不做复杂跨域重算
- 目标响应：秒级

### L1 准实时主题层

用于趋势、排行、汇总分析：

- 月度收款趋势
- 客户欠款排行
- 供应商采购金额排行
- 仓库周转趋势
- 报花趋势、任务分布趋势

特征：

- 轻量主题表、预聚合或缓存
- 刷新周期 1 到 5 分钟
- 目标响应：1 到 3 秒

### L2 展示模板层

用于统一展示资产：

- 报表目录
- 仪表盘卡片
- 大屏组件
- 报告工厂章节模板
- Copilot 模板命中结果

## 统一模板模型

每个固定报表以 `ReportTemplate` 为唯一资产：

- `templateCode`
- `name`
- `domain`
- `category`
- `dataSourceType`
- `targetObject`
- `sqlTemplate`
- `parameterSchema`
- `metricDefinition`
- `refreshPolicy`
- `permissionPolicy`
- `presentationSchema`
- `certification`

设计原则：

- 先定义报表，再定义它如何展示
- 一套模板，多端复用
- 展示层不重新定义口径

## 模板驱动的四个入口

### 1. 固定报表中心

按域分类：

- 财务
- 采购
- 仓库
- 报花
- 任务
- 项目点
- 运营

### 2. 专题看板

将高频模板编排成 dashboard / screen。

### 3. Copilot 问答

处理顺序：

1. 先查模板
2. 命中固定模板则直接返回
3. 未命中才进入探索模式

### 4. 模板候选池

将高频未知问题沉淀为候选模板，由产品/数据人员审核升级。

## Copilot 探索路径

### 1. 识别伪未知

很多问题只是固定模板的参数变体：

- 本月供应商采购排行
- 上月供应商采购排行
- 华东区域供应商采购排行

这类问题应先映射到现有模板，而不是直接走自由 NL2SQL。

### 2. 真未知才进入探索

探索必须受控：

- 先识别业务域
- 再选允许视图或主题表
- 再注入 join contract
- 再注入同义词和示例 SQL
- 最后生成 SQL

### 3. 结果分级

- `A级` 命中模板，直接返回固定报表
- `B级` 探索成功，结果可信
- `C级` 仅能部分回答
- `D级` 置信度不足，要求澄清

### 4. 回灌机制

记录：

- 问题原文
- 命中的视图/主题表
- SQL
- 用户反馈
- 导出/收藏/复用频次

当达到阈值后进入模板候选池。

## 业务域优先级

### 第一优先级

- 财务相关
- 采购管理

### 第二优先级

- 仓库管理
- 报花管理
- 任务中心

### 第三优先级

- 项目点管理
- 运营管理

## v1 范围

### v1 固定报表包

#### 财务固定报表包

建议首批：

- 应收总览
- 客户欠款排行
- 项目回款排行
- 月度收款趋势
- 待付款清单
- 预支申请状态
- 报销单状态
- 发票/收付款流程状态

#### 采购-仓库联合报表包

建议首批：

- 待采购清单
- 待下单清单
- 在途采购
- 待入库清单
- 库存现量
- 低库存预警
- 供应商采购金额排行
- 采购时效分析

#### 报花-任务样板包

建议作为第二梯队样板：

- 本月加花/换花/减花趋势
- 项目报花排行
- 待处理任务清单
- 超时任务预警

## 技术演进建议

### Phase 1

- 建立模板模型
- 固化财务与采购仓库的权威取数面
- 建报表目录与模板命中路由

### Phase 2

- 补报花与任务样板包
- 建模板候选池与升级流程

### Phase 3

- 扩展项目点和运营管理专题包
- 评估是否需要更标准的数仓分层

## 成功标准

- 固定报表不再依赖 LLM 现场生成 SQL
- 财务和采购固定报表具备稳定口径和秒级响应
- Copilot 对未知问题先模板命中，再探索式兜底
- 高频未知问题能沉淀为新模板
- dashboard / screen / report factory / copilot 使用同一套模板资产

## 一句话结论

`dts-copilot` 的下一阶段目标不应是“让大模型查报表更快”，而应是：

**把系统升级成“固定报表优先、Copilot 探索兜底”的业务分析产品。**
