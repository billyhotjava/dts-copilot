# XYCYL 主数据优先治理设计

## 目标

基于 `adminapi / adminweb / app` 三端源码和 `rs_cloud_flower` 业务库，先建立一套可执行的主数据优先治理框架，再在此基础上推进固定报表与 Copilot 的受控取数面建设，降低当前多人口径、硬编码状态和跨模块流程缠绕带来的维护成本。

## 背景判断

当前系统的主要问题不是“功能不够”，而是“业务沉淀压过了领域抽象”：

- 后端已经按 `project / flowerbiz / tasknew / purchase / storehouse / finace / operate` 分了业务目录，但更像按页面和流程长出来的模块，而不是干净的领域边界。
- 前端已经沉淀了大量报表型页面，但很多报表能力散落在业务菜单里，没有统一目录、统一口径和统一命名。
- 业务库 `rs_cloud_flower` 有 `163` 张物理表、`0` 张视图，说明现网更多依赖业务表直出，而不是稳定的数据消费层。
- 财务、采购、库存、履约等能力已形成深业务覆盖，但很多关键状态仍散落在页面枚举、控制器参数和服务内部逻辑里。

因此，不应先从“大报表平台”或“财务中台”切入，而应先从主数据治理切入。

## 设计原则

### 1. 先锚点，后口径

先统一全系统最关键的业务锚点主数据，再统一报表口径、财务取数面和 Copilot 语义层。

推荐的锚点顺序：

1. 项目轴主数据
2. 物品轴主数据
3. 共享参考主数据
4. 财务参考主数据

### 2. 不把交易事实误当主数据

以下对象属于交易事实、状态事实或派生事实，不应作为底层主数据治理对象：

- 库存余额
- 月账单 / 对账单
- 采购汇总
- 任务项
- 报花单
- 凭证
- 项目实摆明细

这些对象应基于主数据和业务事实生成，而不是反过来成为主数据来源。

### 3. 主数据先服务“稳定识别”，再服务“全量重构”

第一阶段目标不是推翻现有表结构，而是建立：

- 唯一命名
- 唯一归属
- 唯一识别键
- 停用 / 启用规则
- 别名 / 映射规则

让后续固定报表、指标模板、Copilot 先有稳定挂点。

## 方案比较

### 方案 A：从财务主数据先做

优点：

- 对管理层感知最强
- 最容易跟“报表治理”直接挂钩

缺点：

- 当前财务更多是业务财务，不是独立总账体系
- 财务口径依赖项目、采购、报花、结算流程本身
- 容易只治理末端口径，不治理上游对象

### 方案 B：从项目主数据单轴先做

优点：

- 与现网主线高度一致
- 项目点确实是大部分业务的载体

缺点：

- 如果不同时治理物品轴，采购、库存、报花、实摆仍会继续分裂

### 方案 C：双锚点主数据优先

推荐方案。

核心做法：

- 先做项目轴主数据：客户、合同、项目点、楼层、摆位、项目角色
- 再做物品轴主数据：物品、分类、属性、规格、价格、项目别名
- 再补库房、供应商、人员组织、字典和财务参考数据

原因：

- 系统大多数业务事实都挂在“项目点 × 物品”两个轴上
- 采购、库存、报花、任务、结算都可以围绕这两个轴稳定识别
- 这是后续固定报表和 Copilot 消费面的最小稳定骨架

## 主数据分层设计

### P0：核心锚点主数据

#### 项目轴

- 客户
- 合同
- 合同部门 / 结算配置
- 项目点
- 楼层 / 楼号 / 摆位
- 项目角色 / 项目负责人

治理目标：

- 一个项目点在全系统只有一个稳定身份
- 合同、客户、项目点、结算配置之间的关系唯一清晰
- 现场任务、报花、采购、账单都能稳定回挂到项目点

#### 物品轴

- 物品
- 分类
- 属性
- 规格
- 价格
- 物品别名 / 项目别名

治理目标：

- 物品编码、规格、价格口径不再在采购、库存、项目、报花里各自演化
- “同物异名”与“项目内部别名”可以被系统识别，而不是靠人工记忆

### P1：共享参考主数据

- 库房
- 供应商
- 员工 / 组织 / 角色
- 状态字典
- 来源码 / 类型码 / 结算方式 / 付款方式

当前特别问题：

- 供应商在现系统中更像挂在业务页面上的信息，不是清晰的一等主数据
- 大量状态和来源码硬编码在前端页面和后端服务中

治理目标：

- 让采购、库存、财务、流程审批统一使用同一套参考字典
- 让组织范围、项目范围、角色范围可被固定报表和 Copilot 正确消费

### P2：财务参考主数据

- 科目
- 对账周期
- 发票类型
- 结算方式
- 冲抵类型
- 付款类型

注意：

这层不宜先做。应在项目轴和物品轴稳定后再做，否则财务口径仍然会缺少业务锚点。

## 主数据与交易事实边界

需要把现有对象重新划边界：

### 保持为主数据

- customer
- contract
- project
- position / floor / floor_number
- goods / goods_price / goods_attribute / goods_classify
- storehouse
- supplier
- org / user / role / dictionary

### 降级为业务事实 / 配置事实

- project_green
- contract_rent_fee
- stock_info balance
- purchase_info
- month_account
- voucher
- flower_biz
- daily_task

其中：

- `project_green` 和 `contract_rent_fee` 更接近“配置型业务事实”
- `stock_info` 应作为库存结果面，不应反向成为物品主数据源

## 与固定报表、Copilot 的关系

主数据优先治理的直接收益不是“马上重写页面”，而是先为两类消费面提供稳定基础：

### 固定报表

- 财务、采购、库存、项目汇总页可以统一挂到项目轴 / 物品轴 / 共享字典上
- 统计口径不再依赖页面自己的枚举和 if/else

### Copilot

- 自然语言里的“项目点 / 物品 / 供应商 / 库房 / 合同 / 客户”可以先映射到主数据
- 再决定落到固定报表模板还是探索式 SQL

## 第一阶段交付范围

第一阶段不直接改老业务流程，只完成以下 5 件事：

1. 三端主数据盘点与归属审计
2. 项目轴和物品轴的 canonical model
3. 库房 / 供应商 / 组织 / 字典的参考主数据方案
4. 主数据与交易事实的边界表
5. 固定报表 / Copilot 的主数据消费约束

## 验收标准

- 明确每类主数据的唯一归属模块
- 明确每类主数据的稳定业务键和展示名
- 明确哪些对象不能再被视作主数据
- 明确首批固定报表模板和 Copilot 问答应挂载哪些主数据轴
- 形成下一阶段可执行 sprint，而不是停留在分析结论
