# 分析工作台双入口协同设计

## 背景

`查询` 菜单页与 `AI Copilot` 现在已经能通过 `analysis_draft` 打通基础工作流，但整体产品体验还停留在“链路可用”，没有进入“工作台一致”阶段。

当前问题主要有四类：

1. 各页面都在各自渲染“来自分析草稿/固定报表”的横幅，样式、动作和语义不一致。
2. `查询` 页面已经能承接草稿，但还没有真正成为“分析资产中心”。
3. `Copilot` 和菜单页虽然平级，但用户在二者之间往返时，状态与来源信息仍然容易断裂。
4. 草稿虽然能晋升到查询、仪表盘、大屏、报告工厂，但这些页面还没有统一的“来源感知”和“后续动作”。

## 目标

1. 保持 `查询` 与 `AI Copilot` 平级入口不变。
2. 让平级入口共享同一套“分析资产工作台”语义。
3. 将来源上下文、状态、后续动作组件化，而不是分散在每个页面里。
4. 让 `analysis_draft` 成为从对话到正式分析资产的统一起点。
5. 让用户在 `Copilot -> 查询 -> 可视化 -> 仪表盘/大屏/报告` 之间切换时，始终能看到清晰的来源与下一步。

## 核心判断

### 1. 入口平级，但“工作台层”必须统一

导航可以继续保留：

- `查询`
- `AI Copilot`

但二者不应各自维护一套临时状态。  
更合理的模型是：

- `Copilot` 负责生成和解释
- `查询` 负责管理和晋升
- `仪表盘 / 大屏 / 报告` 负责消费和展示

三者共享同一层来源语义。

### 2. 来源横幅不应继续分散实现

当前 `CardEditorPage`、`DashboardEditorPage`、`ReportFactoryPage`、`ScreensPage` 都在单独拼自己的来源 UI。  
这会导致：

- 文案漂移
- 动作不一致
- 后续新增 surface 时继续复制代码

因此需要抽象成统一的：

- `analysis provenance model`
- `AnalysisProvenancePanel`

### 3. 分析资产中心应比“草稿列表”更完整

`查询` 页面下一阶段不只是显示草稿，还要成为统一的分析资产中心：

- 全部
- 正式查询
- Copilot 草稿
- 最近分析

并支持：

- 来源筛选
- 状态筛选
- 草稿归档/删除
- 从草稿回到对话

### 4. 多端晋升要强调“来源保留”

仪表盘、大屏、报告工厂不应只知道自己“是新建的”，还应知道：

- 来自哪条问题
- 来自哪份草稿
- 来自哪个查询卡片

这样后续才能支持：

- 回跳到查询
- 追溯口径
- 审计和治理

## 产品结构

### 工作台分层

1. `Copilot`
   - 自然语言生成层
2. `分析草稿`
   - 临时分析资产层
3. `正式查询`
   - 可复用查询资产层
4. `消费面`
   - 仪表盘
   - 大屏
   - 报告工厂

### 统一来源上下文

统一来源对象抽象为：

- `analysisDraft`
- `savedQuery`
- `fixedReport`

每个消费面都通过相同的上下文构建规则来决定：

- 标题
- 说明
- 来源标签
- 推荐动作

## 实施策略

### Phase 1

- 抽取 `analysis provenance model`
- 抽取统一 `AnalysisProvenancePanel`
- 替换各页面的散乱横幅实现

### Phase 2

- 深化 `查询` 页面为分析资产中心
- 支持来源/状态过滤与最近分析视图
- 增强草稿生命周期动作

### Phase 3

- 加强 `Copilot <-> 查询` 双向往返
- 从查询页回跳对话
- 同步草稿、查询和多端资产状态

### Phase 4

- 打通草稿到可视化直达
- 在仪表盘/大屏/报告中保留来源追溯链

## 验收标准

1. 四个主要页面使用统一的来源上下文组件。
2. `查询` 页面具备完整的分析资产中心行为，而不只是草稿承接页。
3. `Copilot` 与菜单页之间的状态和来源不再断裂。
4. 多端资产能追溯到草稿和正式查询来源。
5. `it/` 中存在可重复验证的 smoke 资产。
